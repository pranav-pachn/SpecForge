# 🏗️ Architecture

This document outlines the technical architecture, data model, and core workflows of SpecForge.

## System Overview

SpecForge is built as a monolithic web application using **Next.js 14 App Router**. 
It leverages server-side rendering and server components for fast initial loads, with client components for interactive UI elements.

- **Frontend**: React, TailwindCSS, Lucide Icons
- **Backend**: Next.js Route Handlers & Server Actions
- **Database**: PostgreSQL
- **ORM**: Prisma
- **Auth**: Auth.js (NextAuth v5)
- **AI**: Vercel AI SDK + OpenAI

## Data Model

The database schema is designed to represent the hierarchical nature of a software project workflow and the lifecycle of artifacts within it.

### Core Entities

- **`User` / `Workspace` / `Project`**: Standard SaaS hierarchy. Users belong to workspaces, which contain projects.
- **`Workflow`**: The central entity representing a feature being developed (e.g., "Dark Mode Toggle"). A workflow progresses through multiple statuses (`DRAFT`, `SPEC_REVIEW`, `PLANNING`, `EXECUTING`, etc.).
- **`Artifact`**: A document produced during the workflow (e.g., SPEC, PLAN, TASK_BREAKDOWN). 
- **`ArtifactVersion`**: Artifacts are versioned. Each version has a status (`DRAFT`, `NEEDS_REVIEW`, `APPROVED`, `STALE`, `SUPERSEDED`).
- **`Task`**: Granular execution steps derived from a Plan. Tasks can have dependencies on other tasks.
- **`DriftEvent`**: Automatically generated when an upstream artifact changes (e.g., Spec is updated), marking downstream artifacts (e.g., Plan, Tasks) as `STALE`.

### The Versioning & Drift Engine

A key architectural feature of SpecForge is the **Drift Engine**. 
Since AI agents need up-to-date context, when a user changes requirements (modifies the Spec), the system must invalidate downstream work.

1. When a new `ArtifactVersion` of type `SPEC` is approved:
2. The system checks for any downstream `ArtifactVersion` of type `PLAN` or `TASK`.
3. If they exist, it creates a `DriftEvent` linking the new Spec version to the downstream artifacts.
4. The downstream artifacts' statuses are updated to `STALE`, notifying the user that they need to regenerate their plan/tasks.

## AI Integration

SpecForge uses the **Vercel AI SDK** with OpenAI (`gpt-4o` or similar models) for generation tasks.
AI interactions are encapsulated in Route Handlers under `/api/ai/`.

- `/api/ai/spec`: Takes raw user input and structured fields, generates a 12-section PRD.
- `/api/ai/plan`: Takes an approved Spec and generates a technical architecture and sequencing plan.
- `/api/ai/tasks`: Decomposes an approved Plan into granular Jira-like tasks.

**Prompts** are centrally managed in `src/lib/ai/prompts.ts` to ensure consistent formatting and strict adherence to the output templates required by the application.

## API Design

The application uses standard Next.js App Router API routes (`src/app/api/...`) for RESTful interactions.
- `/api/workflows`: Create and list workflows.
- `/api/artifacts`: Manage documents within a workflow.
- `/api/events`: Lightweight event logging for instrumentation.

## Styling & UI Components

- The UI uses vanilla **TailwindCSS**. 
- Typography is handled by `@tailwindcss/typography` to beautifully render markdown content generated by the AI.
- Animations utilize lightweight CSS keyframes defined in `globals.css` (e.g., `animate-in`, `animate-up`, `animate-pulse`) rather than heavy animation libraries to keep the bundle size minimal and performance high.
